# 演示视频脚本 — 砚台（网络小说智能生成平台）

> **总时长**：约 8-10 分钟  
> **结构**：Part 1 Agent 应用（产品展示，4-5min）+ Part 2 实现架构（技术讲解，3-4min）  
> **录制建议**：1080p，前端界面全屏录制，关键架构部分可用 Keynote/PPT 配合录屏

## Part 2 — 实现架构介绍（技术讲解风格）

> **目标**：向技术评审展示系统架构的深度、Agent 设计的合理性、关键技术的创新点。  
> **情绪基调**：专业、清晰、自信，突出设计决策而非简单堆叠技术名词。

---

### 【镜头 6】总体架构概览（1min）

| 项目 | 内容 |
|------|------|
| **画面** | PPT 展示总体架构图（6 层）：Client Layer（React + Ant Design X）→ Gateway（Nginx，可选）→ Application Layer（FastAPI）→ Services Layer（ContentService Facade + 领域服务）→ AI Engine Layer（Orchestrator + Sub-Agents + Tool Factory）→ Persistence Layer（SQLite + Markdown Files + Checkpoint）。每层用不同颜色区分。底部标注："单机部署 · 无需云服务器"。 |
| **旁白** | "这是系统的总体架构。前端基于 React 18 和 Ant Design X 构建，后端是 FastAPI 服务，全部运行在本地。这是一个单机部署项目——创作者在自己的电脑上安装，数据完全本地存储，无需云服务器，也无需联网（除调用 LLM API 外）。后端的 AI 编排引擎是核心，基于 LangGraph 的多智能体协作系统。持久层采用混合存储：SQLite 存元数据，Markdown 文件存创作内容本体，LangGraph Checkpoint 存会话状态，全部落在本地磁盘。" |
| **时长** | 1min |

### 【镜头 7】多 Agent 协作架构（1min 30s）

| 项目 | 内容 |
|------|------|
| **画面** | PPT 切换为 Agent 架构图。中心是 Orchestrator（Supervisor），外围 5 个子 Agent：world_builder、character、plot、chapter、review。箭头标注 Handoff 关系。下方标注："Subagents-as-Tools Pattern"。 |
| **旁白** | "AI 引擎层采用 Subagents-as-Tools 模式。Orchestrator 是一个 ReAct Agent，它通过系统提示词中定义的 5 个 handoff 工具——delegate_to_world_builder、delegate_to_character 等——来决定把任务路由给哪个子 Agent。每个子 Agent 本身也是一个独立编译的 LangGraph 图，拥有专属的系统提示词和工具集。" |
| **画面（续）** | 动画演示一次完整调用：User Input → Orchestrator 思考"这是世界观相关请求" → 调用 delegate_to_world_builder → 子 Agent 执行 query_content → get_content → create_world_setting → 返回结果 → Orchestrator 汇总回复用户。 |
| **旁白（续）** | "为什么不用单一 Agent？因为小说创作涉及高度专业化的领域知识。让世界观专家去写打斗场面，或者让写手去设计魔法体系，都不合理。多 Agent 架构让每个专家做自己擅长的事，Orchestrator 负责统筹协调——这正是人类出版社的分工模式。" |
| **时长** | 1min 30s |

### 【镜头 8】三大关键技术决策（1min 30s）

| 项目 | 内容 |
|------|------|
| **画面** | PPT 分三栏，每栏一个技术点，配简图。 |
| **旁白** | "这里有三个支撑整个系统的关键技术决策，我想重点展开。" |

**技术点 1：闭包绑定 Project ID（20s）**
| **画面** | 左侧栏放大，显示代码示意图：`def create_world_tools(project_id): @tool def create_world_setting(name, content): ... asyncio.run(_create())`。红色高亮 `project_id` 在闭包中被捕获。 |
| **旁白** | "第一，项目隔离。所有工具函数通过工厂模式创建，project_id 通过 Python 闭包绑定到每个工具内部。这意味着：没有模块级单例，不存在线程安全问题，更不可能出现项目 A 的工具误操作项目 B 的数据。" |

**技术点 2：SSE v2 Streaming + 子图事件透传（40s）**
| **画面** | 中间栏放大，显示 SSE 事件流图。backend `graph.astream(..., stream_mode=["messages","updates","custom"], subgraphs=True, version="v2")` → 前端 EventParser 分类处理 → 时间线渲染。标注不同事件类型的颜色：message（蓝）、tool_start/end（橙）、thinking（紫）、error（红）。 |
| **旁白** | "第二，状态可视化。后端使用 LangGraph v2 Streaming API，开启 subgraphs=True，这样子 Agent 内部的事件也能透传到前端。前端的时间线不是伪造的进度条，而是真实的事件映射：on_chat_model_stream 变成 token 流，on_tool_start/end 变成工具节点，Claude 的 thinking 内容也会实时展示。用户看到的'正在查询世界观…'，就是 Agent 此刻正在做的事。" |

**技术点 3：SkillMiddleware 动态技能注入（30s）**
| **画面** | 右侧栏放大，显示 Skill 模板格式（YAML Frontmatter + Markdown 正文），以及 SkillMiddleware 在模型调用前自动拼接技能描述的流程图。 |
| **旁白** | "第三，技能系统。创作者可以把自己的写作方法论沉淀为 Skill 模板——比如'黄金三章写作法'——以 Markdown + YAML 格式保存。SkillMiddleware 在每个 Agent 调用 LLM 前，自动将匹配 domain 的技能描述注入系统提示词。更关键的是，Orchestrator 还具备运行时创建新技能的能力，Agent 在创作过程中发现有效的方法论，可以自动沉淀下来供未来复用。" |
| **时长** | 1min 30s |

### 【镜头 9】错误处理、持久化与扩展性（1min）

| 项目 | 内容 |
|------|------|
| **画面** | PPT 快速展示三张图：① AIError 分类映射表（RateLimit/Timeout/Network→重试；Auth/BadRequest→立即报错）；② 指数退避公式 `delay = base_delay * 2^(attempt-1) * jitter`；③ Checkpoint 恢复流程（会话中断 → 重启 → aget_state → 恢复上下文）。 |
| **旁白** | "可靠性方面，我们实现了完整的错误分类和恢复机制。LLM 异常被映射为结构化的 AIError，可重试错误自动进行指数退避重试，不可重试错误通过 SSE error 事件实时通知前端。会话状态通过 LangGraph Checkpoint 持久化到 SQLite，即使服务器重启，用户回到聊天也能无缝续聊。" |
| **画面（续）** | 出现 Roadmap：Human-in-the-Loop（关键写操作人工确认）→ LangSmith 全链路追踪 → 多 LLM 供应商支持（OpenAI/本地模型）→ 可选 SaaS 化部署。 |
| **旁白（续）** | "目前的架构已经为未来扩展预留了充足空间：HITL 人工审批节点、LangSmith 追踪、多模型供应商切换。当然，如果未来要从单机工具扩展为云服务，只需要把 SQLite 替换为 PostgreSQL，加上用户认证层——核心 Orchestrator 逻辑完全不用动。" |
| **时长** | 1min |
