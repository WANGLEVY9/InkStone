滑动窗口算法详情

2.1 算法伪代码

# Smart Context Truncation + Sliding Window Attention # 输入: 前文背景 text, 用户问题 query # 输出: 截断后的上下文 context function smart_truncate(text, query, max_chars=300): # Step 1: 如果文本长度小于阈值，不做处理 if len(text) <= max_chars: return text # Step 2: 提取文本头部（包含初始设定） head = text[0:max_chars//3] # Step 3: 提取文本尾部（最近事件，与 query 最相关） tail = text[-(max_chars//3 * 2):] # Step 4: 计算中间段与 query 的语义相关性 mid = text[max_chars//3 : -(max_chars//3 * 2)] sentences = split_sentences(mid) scored = [(s, cosine_sim(embed(s), embed(query))) for s in sentences] scored.sort(key=lambda x: x[1], reverse=True) # Step 5: 保留与 query 最相关的中间句子 mid_selected = [] char_count = 0 for sent, score in scored: if char_count + len(sent) > max_chars // 3: break mid_selected.append(sent) char_count += len(sent) mid_selected.sort(key=lambda x: text.index(x)) # Step 6: 拼接并返回 return head + "\n[...]" + "\n".join(mid_selected) + "\n[...]\n" + tail

算法核心设计理念

滑动窗口策略的核心假设是：前文背景的头部（初始设定）和尾部（最近事件）对续写的贡献度最高。中间段则通过语义相似度筛选保留与当前 query 最相关的句子。这种"头 + 语义锚点 + 尾"的三段式结构在 token 节省与信息保留之间取得了良好的平衡。实验表明，在 301-500 字区间，该策略在不损失关键信息的前提下将 faithfulenss 提升了 0.03-0.04。

3.3 延迟改善来源分解

R04 延迟降低的贡献因素

智能截断 (-180ms, 55%)

减少输入序列长度直接降低了 Transformer 自注意力的计算量（O(n²) 复杂度），短序列的推理速度提升最为显著。

滑动窗口 (-90ms, 28%)

通过限制注意力范围为局部窗口，降低了每层的全局注意力计算量，在长序列上效果尤为明显。

缓存优化 (-55ms, 17%)

引入 KV-Cache 复用机制，对重复出现的上下文前缀跳过重复计算。

尚存问题
  • >500 字上下文仍退化：faithfulness 仅 0.62，context_recall 仍有提升空间。
  • 语义筛选的计算开销：向量嵌入与相似度计算增加了预处理延迟 (约 +120ms)。
  • 截断可能丢失伏笔：部分长篇小说中的远距离伏笔被截断工具过滤，影响后续情节的连贯性。
  • 历史与科幻类忠实度偏低：0.56 和 0.59 仍低于 0.60 基线。
下阶段建议
  • 知识注入：为历史/科幻类构建领域知识库，从根本上减少设定冲突。
  • 摘要压缩：对 >500 字的超长上下文引入 LLM 摘要步骤，替代直接截断。
  • 动态参数配置：根据类型和上下文长度动态调整 temperature 和窗口大小。
  • 远距离伏笔追踪：建立专门的"伏笔寄存器"，在截断时优先保留。
